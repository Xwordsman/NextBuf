# NextBuf 第一阶段开发任务

## 1. 阶段目标

第一阶段目标不是做完整社区，而是让 NextBuf 作为一个真实项目跑起来，并具备最小可用社区闭环。

阶段完成后应达到：

- 本地可以启动开发环境。
- Docker Compose 可以启动 Web、PostgreSQL、Redis。
- 首次访问可进入 `/install` 完成站点初始化。
- 可以创建管理员账号并登录后台。
- 可以浏览节点、发帖、回复。
- 可以进入基础后台管理节点、帖子和用户。
- 推送 GitHub 后可以通过 GitHub Actions 构建 Docker 镜像。
- 服务器宝塔 Docker 容器编排可以拉取镜像运行。

第一阶段暂不追求：

- 完整插件市场
- 完整主题市场
- 复杂积分和勋章
- 复杂审核流
- 复杂私信系统
- 第三方搜索服务
- 对象存储
- 多语言
- 高级 SEO

## 2. 技术选择

第一阶段建议直接确定以下技术栈，减少摇摆：

```text
Next.js
TypeScript
Tailwind CSS
shadcn/ui
PostgreSQL
Redis
Drizzle ORM
Docker Compose
GitHub Actions
```

ORM 第一阶段建议选择 Drizzle。NextBuf 后续会有较多列表查询、聚合查询和索引优化，Drizzle 更适合保留 SQL 可控性。

## 3. 任务拆分

### P0：项目初始化

目标：建立可以长期开发的基础工程。

任务：

- 初始化 Next.js + TypeScript 项目。
- 配置 Tailwind CSS。
- 接入 shadcn/ui 基础组件。
- 配置 ESLint、TypeScript 检查和格式化规则。
- 建立基础目录结构。
- 增加 `.env.example`。
- 增加基础 README。

建议目录：

```text
src/app
src/components
src/features
src/lib
src/server
src/db
src/config
src/styles
```

验收标准：

- `npm run dev` 可以启动。
- 首页可以正常访问。
- TypeScript 检查通过。

### P1：数据库与基础模型

目标：建立最小社区闭环需要的数据结构。

任务：

- 接入 PostgreSQL。
- 接入 Drizzle ORM。
- 建立数据库迁移机制。
- 定义基础数据表。
- 增加数据库连接健康检查。

第一阶段数据表：

```text
users
sessions
site_settings
nodes
posts
replies
tags
post_tags
themes
plugins
```

说明：

- `themes` 和 `plugins` 第一阶段只做数据预留，不开发完整系统。
- `site_settings` 用于判断是否已安装。
- `sessions` 可根据最终认证方案调整。

验收标准：

- 可以执行数据库迁移。
- 数据库连接失败时有明确错误。
- 基础表结构可被应用读取。

### P2：Docker 本地运行

目标：让项目从第一阶段就接近最终部署方式。

任务：

- 编写 `Dockerfile`。
- 编写 `docker-compose.yml`。
- 编写 `docker-compose.dev.yml` 或开发说明。
- 配置 PostgreSQL 服务。
- 配置 Redis 服务。
- 配置 Web 服务环境变量。

验收标准：

- `docker compose up -d` 可以启动 Web、PostgreSQL、Redis。
- Web 容器可以连接数据库和 Redis。
- 重启容器后数据库数据不丢失。

### P3：首次安装向导

目标：站长第一次启动后可以完成初始化。

任务：

- 实现 `/install` 页面。
- 实现未初始化状态检测。
- 未初始化时访问前台、后台自动跳转 `/install`。
- 安装页填写网站名称、网站网址、管理员用户名、邮箱、密码。
- 创建第一个管理员用户。
- 写入站点初始化状态。
- 安装完成后跳转 `/admin`。
- 已初始化后禁止再次访问安装提交接口。

验收标准：

- 空数据库首次访问进入安装页。
- 安装成功后可以登录后台。
- 已安装站点不能重复安装。
- 管理员密码使用安全哈希存储。

### P4：认证与账号

目标：实现最基本的账号体系。

任务：

- 实现注册。
- 实现登录。
- 实现退出登录。
- 实现当前用户读取。
- 实现管理员权限判断。
- 实现基础用户资料页。

第一阶段账号规则：

- 支持邮箱 + 密码登录。
- 用户名唯一。
- 邮箱唯一。
- 第一个安装用户为管理员。
- 暂不接 OAuth。
- 暂不强制邮件验证。

验收标准：

- 普通用户可以注册和登录。
- 管理员可以访问 `/admin`。
- 普通用户不能访问后台。

### P5：节点基础功能

目标：实现两级节点的最小能力。

任务：

- 实现节点列表页。
- 实现节点详情页。
- 实现一级节点和二级节点。
- 一级节点详情默认聚合自身和子节点帖子。
- 实现后台节点创建、编辑、启用、隐藏。
- 实现发帖时选择节点。

第一阶段可以简化：

- 节点申请暂不做。
- 节点试运行暂不做。
- 节点权限先支持 `open` 和 `admin_only`。

验收标准：

- 可以在后台创建一级节点和二级节点。
- 前台可以浏览节点。
- 发帖时可以选择节点。

### P6：帖子与回复

目标：完成社区最核心闭环。

任务：

- 实现首页帖子列表。
- 实现帖子详情页。
- 实现发帖页面。
- 实现回复功能。
- 实现帖子和回复的基础状态。
- 更新回复数和最后回复时间。

第一阶段内容规则：

- 编辑器先使用 Markdown 或普通 textarea。
- 帖子标题必填。
- 正文必填。
- 回复内容必填。
- 暂不做复杂富文本。
- 暂不做图片上传。

验收标准：

- 登录用户可以发帖。
- 登录用户可以回复。
- 首页可以看到最新帖子。
- 帖子详情可以看到正文和回复。

### P7：基础后台

目标：提供最基本的运营入口。

任务：

- 实现 `/admin` 后台首页。
- 显示站点基础统计。
- 实现用户列表。
- 实现帖子列表。
- 实现回复列表。
- 实现节点管理。
- 实现站点设置页。

第一阶段治理能力：

- 隐藏帖子。
- 恢复帖子。
- 隐藏回复。
- 恢复回复。
- 禁用用户。

验收标准：

- 管理员可以进入后台。
- 管理员可以管理节点。
- 管理员可以隐藏和恢复内容。
- 管理员可以禁用用户。

### P8：主题与插件预留

目标：不开发完整系统，但避免未来重构。

任务：

- 建立 CSS variables 主题 token。
- 内置默认主题 `nextbuf-default`。
- 建立 `themes` 表。
- 建立 `plugins` 表。
- 建立插件 manifest TypeScript 类型。
- 建立基础事件名和 hook 类型定义。

第一阶段不做：

- 在线安装主题。
- 在线安装插件。
- 第三方插件执行。
- 插件市场。
- 主题市场。

验收标准：

- 页面颜色和圆角来自 CSS variables。
- 数据库中可以记录当前主题。
- 代码中存在清晰的插件类型和扩展点命名。

### P9：GitHub Actions 与镜像发布

目标：打通 GitHub 到服务器部署的关键路径。

任务：

- 增加 GitHub Actions workflow。
- 执行依赖安装。
- 执行类型检查。
- 执行构建。
- 构建 Docker 镜像。
- 推送到镜像仓库。
- 编写宝塔 Docker 容器编排部署说明。

镜像标签：

```text
latest
sha-<commit>
```

验收标准：

- 推送主分支后自动构建镜像。
- 镜像可以被服务器拉取。
- 使用生产 `.env` 可以启动容器。

### P10：基础测试与文档

目标：保证第一阶段可交付、可部署、可继续开发。

任务：

- 增加基础 smoke test。
- 增加安装流程测试。
- 增加认证基础测试。
- 增加部署文档。
- 增加环境变量说明。
- 增加常见问题说明。

验收标准：

- 新开发者可以根据 README 启动项目。
- 站长可以根据部署文档完成容器编排。
- 安装、登录、发帖、回复主流程可验证。

## 4. 第一阶段推荐开发顺序

```text
P0 项目初始化
P1 数据库与基础模型
P2 Docker 本地运行
P3 首次安装向导
P4 认证与账号
P5 节点基础功能
P6 帖子与回复
P7 基础后台
P8 主题与插件预留
P9 GitHub Actions 与镜像发布
P10 基础测试与文档
```

## 5. 第一阶段最小上线标准

满足以下条件即可认为第一阶段完成：

- 新环境启动后能进入安装页。
- 安装后能创建管理员。
- 管理员能登录后台。
- 管理员能创建节点。
- 用户能注册登录。
- 用户能发帖和回复。
- 首页、节点页、帖子详情页可正常访问。
- Docker Compose 可以运行。
- GitHub Actions 可以构建镜像。
- 宝塔 Docker 容器编排可以拉取镜像运行。

## 6. 暂缓到第二阶段

以下内容不要塞进第一阶段：

- 举报完整流程
- 审核队列
- 节点申请
- 信任等级自动升级
- 通知系统
- 点赞
- 收藏
- 搜索
- 图片上传
- 邮件发送
- OAuth 登录
- 插件市场
- 主题市场
- 对象存储
- Redis 高级缓存策略

第一阶段的原则是先让项目真实跑起来。只要主流程稳定，后面治理、体验、扩展能力都可以一层层加。
